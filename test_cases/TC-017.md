# TC-017 — Check display of search results when updating search location

- **ID:** TC-017
- **Title:** Check display of search results when updating search location (re-search)
- **URL(s) under test:** https://www.travala.com/search
- **Preconditions:** User is on the search results page with results loaded (run TC-010 as setup; baseline location = Hanoi, Vietnam).
- **Note:** Source steps were high-level ("Change location → Click Search again → Verify results"). The new location and the definition of "results update" below are concrete interpretations. [ASSUMED] New location: **Da Nang, Vietnam** — same country, unambiguous city, chosen only to make the test executable; any second city works.

## Steps

1. [ACTION] Record the current URL `location` param value and the current "N properties found" count (baseline). [ASSUMED — needed to verify "results update".]
2. [ACTION] Confirm the destination input in the results-page search bar is visible; if not after 5 s, fail the step. [ASSUMED — element-existence guard.]
3. [ACTION] Click the destination input.
4. [ACTION] Clear the existing value ("Hanoi, Vietnam"). [ASSUMED — field hygiene.]
5. [ACTION] Type `Da Nang`.
6. [ACTION] Wait for the autocomplete dropdown to populate. Fallback: 5 s (KB §8).
7. [ACTION] Click the "Da Nang, Vietnam" (**City**) entry.
8. [ACTION] If the date picker auto-opens (KB §2 cascade), close it or leave dates unchanged — do not modify dates. [ASSUMED — ambient cascade handling; dates stay as baseline.]
9. [ACTION] Click **SEARCH**.
10. [ACTION] Wait until the URL `location` param differs from the baseline value AND at least one hotel card is rendered. Fallback: 10 s (KB §8).
11. [CHECKPOINT] Destination field on the results page shows `Da Nang, Vietnam`. Fail → stop and report.
12. [CHECKPOINT] URL `location` param ≠ baseline value; `check_in`/`check_out` params unchanged from baseline. Fail → stop and report full URL.
13. [CHECKPOINT] "N properties found" shows N > 0 and the result set changed: first hotel card differs from the baseline first card, and/or the property count differs. [ASSUMED — concrete meaning of "results update per new location"; identical count AND identical first card = fail.] Fail → stop and report both counts and first-card names.
