# TC-022: Verify discount badge math against strikethrough vs. actual price (Apricot Hotel)

**URL(s) under test:** https://www.travala.com/search (specific query TBD — see step 1 flag)
**Priority:** (not specified in source)

**Preconditions:**
- Search results loaded with the "Apricot Hotel" card visible.

**Failure handling (default):** Stop and report on any failed CHECKPOINT, unless noted otherwise.

## Steps

1. **[ACTION]** Ensure a search results page is loaded that includes the "Apricot Hotel" card — either by continuing from a prior search in this suite, or, if the hotel isn't present in that result set, by searching for "Apricot Hotel" directly in the destination/property search field. **[FLAGGED — reason: the source precondition assumes "Apricot Hotel card visible" but doesn't specify which destination/dates surface it; cannot be inferred concretely, so the agent should fall back to a direct name search and report if the hotel still can't be located.]**
2. **[ACTION]** If Apricot Hotel is not visible in the initial viewport, scroll through the results page to locate it. **[ASSUMED — reason: "Element existence" checklist item; results may be paginated or below the fold.]**
3. **[CHECKPOINT]** Confirm the "Apricot Hotel" card is visible on the results page. If not found after a reasonable scroll/search, stop and report that the test cannot proceed (fail this step). **[ASSUMED — reason: explicit failure behavior for a not-found precondition.]**
4. **[ACTION]** Read and record the discount badge text/percentage shown on the Apricot Hotel card (e.g., "X% OFF").
5. **[ACTION]** Read and record the strikethrough (original/reference) price shown on the card, including its unit basis if labeled (e.g., per night vs. total).
6. **[ACTION]** Read and record the actual (current) price shown on the card, including its unit basis if labeled.
7. **[ACTION]** If the strikethrough and actual prices are not both labeled with the same unit basis, attempt to determine each price's basis independently (e.g., dividing a total by the number of nights in a known-length stay); if the basis genuinely cannot be determined, record this as a data-quality limitation rather than guessing. **[ASSUMED — reason: needed to make the comparison in step 8/9 meaningful; this is exactly the ambiguity this test is designed to surface, per the source's own historical bug note about apples-to-oranges comparison.]**
8. **[ACTION]** Compute `(strikethrough price − actual price) / strikethrough price × 100`, using same-unit-basis prices from step 7.
9. **[CHECKPOINT]** Confirm the computed percentage (step 8) matches the badge percentage (step 4) within ±1 percentage point. Fail if they don't match, and in the report include: the badge percentage, both prices with their unit basis, and the computed percentage. **[ASSUMED tolerance — reason: "Assertion specificity" checklist item; a small rounding tolerance avoids false failures from display rounding.]**
